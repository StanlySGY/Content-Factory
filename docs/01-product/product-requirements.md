## 2. 产品目标

Content Factory 的目标是构建一个工程化 AI 内容生产系统，将基于 Claude Code、Codex、Gemini、OpenCode 等 Agent 的内容生产过程，从一次性聊天协作升级为可配置、可追踪、可复用、可审查、可扩展的内容工厂。

### 2.1 核心目标

- 将内容需求转化为结构化任务、上下文包和可执行工作流。
- 支持多个 Agent 按角色分工完成研究、规划、撰写、审查、修订和发布准备。
- 沉淀可复用的内容模板、工作流、插件、Skill、MCP 工具和项目文档。
- 保证内容生产全过程具备状态追踪、版本记录、质量门禁和人工确认机制。
- 通过清晰模块边界，确保 Agent、MCP、Skill、插件与业务规则解耦。

### 2.2 成功标准

- 用户可以从一个内容需求创建完整生产任务。
- 系统可以按预设工作流调度不同 Agent 完成阶段性产出。
- 每个阶段的输入、输出、状态、责任方和审查结论可追踪。
- 内容资产、提示词、上下文、审查记录和最终稿可复用。
- 新 Agent、新 MCP 工具、新 Skill 可以通过插件化方式接入。

### 2.3 量化成功指标

§2.2 为定性成功标准，本节给出可度量指标。数值为 MVP 设定基线，Beta 阶段依实际数据校准。

| 指标 | 定义 | MVP 目标 |
| --- | --- | --- |
| 北极星：周成稿数 | 每周经平台完成并通过审查的内容成稿数 | 建立基线并持续增长 |
| 任务完成率 | 进入执行的任务中达到 `completed` 的比例 | ≥ 80% |
| 一次通过率 | 初稿首次质量审查即通过的比例 | ≥ 50% |
| 平均修订轮次 | 单篇内容从初稿到最终稿的审查退回次数 | ≤ 1.5 |
| 端到端周期 | 从需求确认到最终稿的中位时长 | ≤ 1 个工作日 |
| 过程可追溯率 | 阶段产出具备来源、版本与审查记录的比例 | 100%（硬性） |
| 扩展达成 | 新增 Agent/MCP/Skill/插件接入是否零业务代码改动 | 是（硬性） |

过程可追溯率与扩展达成为硬性约束，不达标视为未满足产品目标；其余为效率指标，用于迭代优化。

## 3. 产品定位

Content Factory 是面向内容生产、内容运营和 AI Agent 编排的工作台。

它不是单一聊天工具，也不是单一内容编辑器，而是将内容生产流程中的需求管理、资料研究、任务拆解、Agent 协作、内容生成、质量审查和发布准备组织为可持续运行的系统。

### 3.1 产品边界

包含：

- 内容生产任务管理。
- Agent 角色与任务编排。
- 工作流定义与执行追踪。
- 内容资产、上下文、模板和审查记录管理。
- MCP、Skill、插件的接入与治理。
- 内容生产过程中的质量门禁和人工确认。

不包含：

- 替代所有第三方发布平台。
- 将业务规则写死到某个 Agent 或 Prompt 中。
- 直接绑定单一模型供应商或单一 Agent 工具。
- 在无授权情况下自动调用生产环境 API 或外部发布渠道。

**MVP 聚焦**：首要渠道为微信公众号图文，首要内容类型为图文文章（`article`）；`post`/`script` 等内容类型与其他发布渠道经可扩展机制延后接入（见 §4.5、§7、§9.2）。

### 3.2 竞品与差异化

| 竞品类别 | 代表形态 | 局限 |
| --- | --- | --- |
| 通用 AI 对话 | 单轮或多轮聊天助手 | 流程不可复用、状态不可追踪、产出散落聊天记录 |
| AI 写作工具 | 模板化写作与续写 | 缺多阶段编排、缺质量门禁与审查追溯 |
| 内容管理与排期 | 编辑与发布排期工具 | 不承载 AI 生产过程，缺 Agent 协作 |
| Agent 编排框架 | 通用工作流或 Agent 平台 | 偏通用编排，缺内容生产领域模型与审查闭环 |

差异化价值主张：

- **工程化内容工厂**：把一次性聊天升级为可配置、可追踪、可复用的工作流与领域模型。
- **Agent 可替换**：Claude Code、Codex、Gemini、OpenCode 经统一抽象接入，不绑定单一供应商。
- **文档驱动 + 业务解耦**：业务规则沉淀于领域层与文档，不写死在 Agent、Prompt 或工具中。
- **质量门禁与全程可追溯**：阶段产出、版本、审查结论可追溯，支撑复盘与持续优化。
- **可治理的扩展**：MCP、Skill、插件经注册与权限治理接入，扩展不破坏核心流程。

## 4. 目标用户

### 4.1 内容负责人

负责制定内容策略、确认内容方向、审批最终内容。

核心诉求：

- 快速创建内容需求。
- 查看内容生产进度。
- 审批关键产出。
- 复用高质量内容流程。

### 4.2 内容运营人员

负责日常选题、素材整理、内容生成、排期和发布准备。

核心诉求：

- 批量管理内容任务。
- 使用模板和工作流提升效率。
- 追踪每篇内容的来源、版本和审查记录。
- 减少重复沟通和手工整理。

### 4.3 AI 工作流设计者

负责设计 Agent 分工、工作流阶段、质量门禁和工具接入方式。

核心诉求：

- 配置可复用工作流。
- 定义 Agent 职责、输入输出和审查规则。
- 接入 MCP 工具、Skill 和插件。
- 调试工作流执行过程。

### 4.4 开发者与系统维护者

负责系统实现、扩展、部署和运维。

核心诉求：

- 清晰理解模块边界和接口契约。
- 安全接入外部 Agent、MCP 和插件。
- 维护可测试、可扩展、可观测的系统。
- 避免业务逻辑和外部工具强耦合。

### 4.5 用户优先级与目标市场

| 优先级 | 用户 | 在 MVP 中的角色 |
| --- | --- | --- |
| 主用户 | 内容运营人员、内容负责人 | 创建需求、推进生产、审批产出的核心日常使用者 |
| 次要用户 | AI 工作流设计者 | 配置工作流、Agent 分工与门禁，低频但关键 |
| 支撑用户 | 开发者与系统维护者 | 接入与维护，不参与日常内容生产 |

- **目标市场**：以公众号图文为首要场景的中文内容生产个人与小团队；MVP 优先服务单项目、单团队规模。
- **首要场景**：围绕主用户的「选题 → 生产 → 审查 → 发布准备」闭环优先打磨，次要与支撑用户的能力随阶段递进开放。
- 主用户决定 MVP 取舍：与主用户核心闭环无关的能力延后。

## 5. 用户场景

### 5.1 从选题创建内容生产任务

用户输入内容主题、目标受众、发布渠道、语气风格和截止时间。系统生成结构化任务，并选择合适工作流。

期望结果：

- 创建内容任务。
- 自动生成需求摘要。
- 绑定默认工作流。
- 明确下一步执行阶段。

### 5.2 多 Agent 协作生产内容

系统将内容任务拆分为研究、提纲、初稿、审查、修订、发布准备等阶段，并分配给不同 Agent。

期望结果：

- 每个 Agent 只处理明确职责范围内的任务。
- 每个阶段生成可审查产出。
- 阶段之间通过结构化上下文传递信息。

### 5.3 人工审查与质量门禁

内容负责人在关键节点查看 Agent 输出，确认是否通过、退回修改或人工编辑。

期望结果：

- 记录审查意见。
- 保留版本差异。
- 阻止未经确认的内容进入发布准备阶段。

### 5.4 复用成熟内容工作流

用户将一条验证有效的内容流程保存为模板，用于后续同类内容生产。

期望结果：

- 保存工作流模板。
- 复用 Agent 分工和阶段定义。
- 支持按不同内容类型调整参数。

### 5.5 接入新 Agent 或工具

系统维护者将新的 Agent、MCP 工具或 Skill 接入平台，并声明能力、权限和失败策略。

期望结果：

- 新能力通过注册机制接入。
- 不修改核心业务流程。
- 可在工作流中按能力选择使用。

### 5.6 追溯内容生成过程

当内容出现质量问题或需要复盘时，用户查看需求、上下文、Agent 输出、审查意见和最终版本。

期望结果：

- 明确内容来源。
- 明确每次修改原因。
- 明确哪个阶段产生问题。
- 支持后续优化工作流。

### 5.7 异常与失败旅程

正向场景之外，系统需对关键失败路径给出明确预期行为：

- **产出质量不达标**：审查未通过时退回上一阶段重做（新建运行、保留血缘），不覆盖历史版本。
- **审查反复退回**：超过约定轮次时升级提示，允许人工接管编辑或终止任务。
- **发布准备失败**：渠道适配或授权校验失败时停在发布准备阶段并记录原因，不产生半发布状态。
- **内容合规风险**：命中合规/版权风险时阻断流转并提示人工确认（见 §9.6）。
- **Agent / 工具失败**：失败可重试或退回，任务状态始终可恢复，中间产物已持久化（见 §9.5）。

## 6. 核心功能

### 6.1 内容任务管理

用于创建、编辑、查看和追踪内容生产任务。

功能要求：

- 创建内容任务，记录主题、目标、受众、渠道、风格、截止时间。
- 管理任务状态：草稿、待执行、执行中、待审查、需修订、已完成、已归档。
- 支持任务优先级、负责人、标签和内容类型。
- 支持任务历史记录和关键事件追踪。

### 6.2 需求结构化

将用户输入的自然语言需求转化为可执行的结构化任务定义。

功能要求：

- 提取内容目标、受众、渠道、格式、约束和验收标准。
- 生成任务摘要和缺失信息提示。
- 支持用户确认或补充需求。
- 解析结果标注置信度；低置信或信息不足时必须提示缺失项并要求人工确认后方可推进，不依赖高精度自动解析。
- 禁止将一次性聊天上下文作为唯一需求来源。

### 6.3 工作流管理

定义和执行内容生产流程。

功能要求：

- 支持工作流模板。
- 工作流包含阶段、输入、输出、执行者、质量门禁和完成标准。
- 支持阶段状态持久化。
- 支持人工暂停、继续、退回和终止。
- 支持工作流执行日志。

### 6.4 Agent 编排

管理 Agent 角色、能力和任务分配。

功能要求：

- 定义 Agent 名称、角色、能力、输入输出和限制。
- 支持按阶段选择 Agent。
- 支持多 Agent 并行或串行协作。
- 支持 Agent 输出审查。
- Agent 不得直接拥有业务规则。

### 6.5 上下文包管理

为 Agent 提供最小必要、可审查的上下文输入。

功能要求：

- 生成阶段上下文包。
- 记录上下文来源和版本。
- 控制敏感信息暴露。
- 支持上下文复用和归档。

### 6.6 内容资产管理

管理内容生产过程中的中间产物和最终产物。

功能要求：

- 保存研究材料、提纲、初稿、修订稿、最终稿。
- 支持版本记录。
- 支持资产标签和检索。
- 支持关联任务、工作流阶段和 Agent 输出。

### 6.7 质量审查

确保内容在进入下一阶段前满足质量要求。

功能要求：

- 支持自动检查和人工审查。
- 审查项包括事实准确性、结构完整性、风格一致性、渠道适配性、风险内容。
- 支持通过、退回、人工修改、终止。
- 审查结果必须记录。

### 6.8 MCP 工具接入

接入检索、浏览器、文件、外部系统等 MCP 工具能力。

功能要求：

- 管理 MCP Server 与工具清单。
- 声明工具权限、输入、输出、超时和错误语义。
- 在工作流阶段中按能力调用工具。
- 对外部网络、敏感数据和生产环境调用执行授权控制。

### 6.9 Skill 管理

管理可复用命令、质量门禁和领域能力。

功能要求：

- 注册 Skill 名称、用途、触发条件和输出格式。
- 支持工作流阶段引用 Skill。
- 支持质量门禁 Skill。
- Skill 不得绕过项目宪法和文档原则。

### 6.10 插件系统

通过插件扩展内容类型、渠道、工具和工作流能力。

功能要求：

- 插件声明名称、版本、能力、权限、输入输出和失败策略。
- 插件通过稳定接口接入。
- 插件不得直接修改核心业务流程。
- 插件失败必须可隔离、可记录、可恢复。

### 6.11 仪表盘与可观测性

提供任务、工作流、Agent、工具调用和质量状态视图。

功能要求：

- 展示任务列表和状态。
- 展示工作流执行进度。
- 展示阶段输出和审查结果。
- 展示 Agent 执行记录和失败原因。
- 展示 MCP 调用记录和风险提示。

### 6.12 发布与渠道管理

管理内容从最终稿到渠道发布的准备、执行与追踪。MVP 首要渠道为公众号图文。

功能要求：

- 配置发布渠道及其适配规则（格式、字数、封面、排版要求）。
- 将最终稿生成渠道适配的发布草稿，公众号图文优先。
- 记录发布动作与结果，锚定具体资产版本（对应 `publish_records`），保证已发布内容不随后续修订漂移。
- 支持发布状态追踪、失败重试、撤回与重发。
- 外部平台调用与发布动作必须显式授权并写入审计。
- 渠道为可扩展能力，新增渠道通过插件接入，不修改核心流程。

## 7. 功能优先级

验收标准：

- 每个阶段的产出可以被记录和审查。
- Agent 角色与业务规则边界清晰。
- 文档覆盖产品、架构、Agent、工作流的基础定义。

### 7.2 P1：核心自动化

- 需求结构化。
- 工作流模板管理。
- Agent 分配规则。
- 上下文包生成。
- 内容资产版本管理。
- 基础质量门禁。

- 用户可以从自然语言需求生成结构化任务。

### 7.3 P2：协作与扩展

- 多 Agent 并行协作。
- 插件注册与隔离。
- Skill 路由与质量门禁自动化。
- 内容模板库。
- 审查规则配置。

验收标准：

- 新 Agent、Skill、MCP 或插件可以在不修改核心业务逻辑的情况下接入。
- 内容生产流程可以按内容类型复用和调整。

### 7.4 P3：规模化与智能优化

- 工作流执行分析。
- 内容质量趋势分析。
- Agent 效果评估。
- 成本、耗时和失败率分析。
- 自动推荐工作流优化方案。
- 团队权限和多租户能力。

验收标准：

- 用户可以基于数据优化内容生产流程。
- 系统可以支持团队级内容工厂运行。

### 7.5 完成定义（DoD）与可测试验收

§7.1~§7.4 的验收标准为方向性描述，本节给出统一完成定义与可测试用例格式，作为各功能项的 Done 判据。

通用完成定义（DoD）：

- 功能满足对应需求项，关键状态流转经领域层校验。
- 设计与变更同步落地到 `docs`，无聊天上下文依赖。
- 具备单元或集成测试覆盖关键路径，且全部通过。
- 关键操作可观测、可审计，失败可恢复。
- 不引入业务规则对 Agent、MCP、Skill、插件的硬耦合。

可测试验收用例（Given-When-Then 示例）：

- 任务创建（P0）：Given 含主题/受众/渠道的需求，When 创建任务，Then 生成结构化任务与需求摘要、绑定默认工作流、状态为 `ready`。
- 阶段产出与审查（P0）：Given 执行中阶段产出初稿，When 审查退回，Then 记录审查意见与版本差异、阶段进入 `revision_required`、不可进入发布准备。
- 需求结构化（P1）：Given 自然语言需求缺失渠道字段，When 结构化，Then 标记缺失项并要求人工确认，不得静默臆测。
- 调用可追溯（P1）：Given 一次 Agent 或 MCP 调用，When 调用完成，Then 输入、输出、状态、耗时可在追溯视图查询。
- 扩展接入（P2）：Given 新增一个 Agent Provider 适配器与配置，When 注册并选用，Then 工作流可调用且未修改核心业务代码。

每个功能项交付前必须满足通用 DoD，并具备至少一条上述格式的可测试验收用例。

## 9. 非功能要求

### 9.1 可维护性

- 核心业务逻辑必须与 Agent、MCP、Skill、插件解耦。
- 模块必须有明确边界和接口契约。
- 文档必须随设计和实现同步更新。

### 9.2 可扩展性

- 新 Agent、新 MCP 工具、新 Skill、新插件必须可注册接入。
- 工作流必须可配置、可复制、可版本化。
- 内容类型和发布渠道必须可扩展。

### 9.3 安全性

- 敏感数据不得默认暴露给 Agent 或 MCP 工具。
- 外部网络、生产环境 API、发布动作必须显式授权。
- 插件和工具调用必须具备权限边界。

### 9.4 可观测性

- 任务状态、阶段状态、Agent 输出、MCP 调用、审查结果必须可追踪。
- 失败必须记录原因、上下文和恢复建议。
- 关键操作必须具备审计记录。

### 9.5 可恢复性

- 工作流必须支持暂停、恢复、退回和终止。
- Agent 或插件失败不得导致任务状态不可恢复。
- 中间产物必须持久化，避免依赖聊天上下文。

### 9.6 合规、版权与数据

- 生成内容须可附合规与事实核查门禁，高风险内容阻断流转并要求人工确认。
- 引用的素材与配图须可追溯来源与授权状态，未授权素材不得进入发布准备。
- 用户数据与生成内容归属用户；数据留存、导出与删除须有明确策略，不默认外泄给 Agent 或外部服务。
