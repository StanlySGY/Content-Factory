## 问题

### Major

#### PROD-001 缺少可量化的成功指标

- 位置：§2.2 成功标准、§7.1 验收标准
- 问题：成功标准与验收标准均为定性陈述（"用户可以……"），无任何可度量指标（如端到端产出周期、单篇人工干预时长、Agent 自动完成率、审核退回率、产出可用率）。
- 影响：无法客观判断 MVP 是否成功，也无法为优先级提供数据依据。

#### PROD-002 验收标准不可测试

- 位置：§7.1 ~ §7.4 各级验收标准
- 问题：验收标准缺少可执行验收用例与边界条件（例如"按阶段推进"未定义哪些阶段、何为完成、失败如何判定）。
- 影响：开发与测试无法据此编写明确验收/E2E 用例，"Done"定义模糊。

#### PROD-003 缺少主要用户与目标市场定义

- 位置：§4 目标用户
- 问题：列出 4 类用户但未指明 MVP 的主要用户（primary persona），未说明面向个人、小团队还是企业，未定义付费方/决策者与市场规模。
- 影响：功能优先级与 UI 取舍缺少锚点，易导致面面俱到、资源分散。

#### PROD-004 缺少竞品分析与差异化价值主张

- 位置：§3 产品定位
- 问题：未分析现有替代方案（直接使用 Claude Code 聊天、通用 CMS、Notion+AI 等），未回答"为何使用 Content Factory 而非直接用 Agent 对话"。
- 影响：价值主张不清，影响产品取舍与采用动机，属产品定义关键缺口。

#### PROD-005 缺少"发布与渠道管理"核心功能项，且与下游设计不一致

- 位置：§6 核心功能（无发布/渠道功能项）对比 `content-workflow.md` §3.9 发布阶段、`ui-design.md` §16 公众号工作台、`development-roadmap.md` Sprint 4 发布准备
- 问题：工作流有"发布"阶段、UI 专门设计了"公众号工作台"、路线图安排了发布准备，但产品功能清单 §6 未列出"发布与渠道管理"功能项。
- 影响：产品功能清单与下游设计不对齐，发布这一关键能力在产品层缺位，易在落地时范围与责任不清。

### Minor

#### PROD-006 内容类型与 MVP 首要渠道未聚焦

- 位置：§6.1（content_type: article/post/script）、§3 产品定位
- 问题：未定义 MVP 支持哪些内容类型与首要渠道。UI 重度投入"公众号工作台"，暗示微信公众号为重点渠道，但产品文档未将其列为首要场景。
- 影响：产品与 UI 在渠道聚焦上不一致，MVP 范围易发散。

#### PROD-007 用户场景缺少异常 / 失败旅程

- 位置：§5 用户场景（全为正向路径）
- 问题：未覆盖 Agent 产出质量不达标、审核反复退回、发布失败、内容合规风险等关键异常旅程。
- 影响：产品对失败路径的预期行为缺失，影响交互与兜底设计。

#### PROD-008 缺少合规、版权与数据留存的产品级要求

- 位置：§9.3 安全性（仅技术视角）
- 问题：产品层未统一规定生成内容合规、配图版权、数据留存与用户数据所有权（配图版权仅零散出现在工作流）。
- 影响：内容生产产品的合规与版权是关键产品约束，缺乏统一要求存在法律与品牌风险。

#### PROD-009 待后续细化链接存在死链

- 位置：§11 待后续细化
- 问题：引用 `system-overview.md`、`data-model.md`、`agent-roles.md`、`tool-contracts.md`、`skill-registry.md`、`content-pipeline.md`、`information-architecture.md`、`api-overview.md`，与实际文件名不符（应为 `system-architecture.md`、`database-design.md`、`agent-architecture.md`、`mcp-architecture.md`、`content-workflow.md`、`ui-design.md`）。
- 影响：与 ARCH-001 同源的系统性命名不一致，破坏文档可导航性。

#### PROD-010 三套阶段术语并存且映射未声明

- 位置：§7 功能优先级（P0-P3）、§8 未来规划（MVP/Alpha/Beta/生产化）对比 `development-roadmap.md`（Sprint 1-4）
- 问题：存在三套阶段划分，未声明其对应关系（P0 是否等于 MVP，是否等于 Sprint 1-4）。
- 影响：团队对"MVP 到底包含什么"理解易产生偏差。

#### PROD-011 需求结构化缺少兜底规则

- 位置：§6.2 需求结构化、§5.1 场景
- 问题：依赖 AI 将自然语言需求解析为结构化任务，但未定义解析不准确或信息不足时的产品行为（缺失项提示、人工补全、置信度展示）的明确规则。
- 影响：MVP 若依赖高精度 AI 解析，鲁棒性不足。

## MVP 修订建议

> 目标：在不扩大范围的前提下，让 MVP 更聚焦、可度量、可验收。仅记录建议，不修改原文档。

1. **聚焦主用户与单一渠道**：MVP 主用户定为单人/小团队内容运营者，首要渠道定为微信公众号图文，单一 content_type，与 UI 公众号工作台对齐。
2. **显式纳入"发布准备"为 P0 功能项**：与路线图 Sprint 4 一致，MVP 仅含"发布准备/渠道适配"，明确不含真实自动发布。
3. **统一阶段术语**：在产品文档声明 P0 = MVP = 路线图 Sprint 1-4，删除或合并 §8 与 §7 的重复阶段划分。
4. **定义 MVP 量化验收**：例如"一篇公众号图文可在系统内从选题走到发布准备""阶段产出 100% 持久化""审核退回后可重新执行"。
5. **显式排除项写入 P0**：在 §7.1 明确 MVP 不含真实多 Agent 自动执行、MCP 市场安装、插件系统、多租户、RAG（与路线图 §2.2 对齐），防止范围蔓延。
6. **需求结构化降级为可控行为**：MVP 仅做"结构化字段抽取 + 缺失项提示 + 人工确认"，不依赖高精度自动解析。

## 最终结论

**结论：有条件通过（Conditional Pass）。**

- 产品功能范围（任务、工作流、Agent、资产、审查、MCP/Skill/插件治理）描述清晰，足以支撑 MVP 启动，无 Critical 阻断项。
- 放行条件（进入深度开发前应处理）：
  1. PROD-003 / PROD-004：补齐主用户与差异化价值（定位根基）。
  2. PROD-005：补齐"发布与渠道管理"功能项，消除与 UI / 工作流的不一致。
  3. PROD-001 / PROD-002：定义可量化指标与可测试验收。
  4. PROD-010 + MVP 修订建议：统一阶段术语并收敛 MVP 范围。
- Minor 问题登记并择机处理。
- 统计：Critical 0，Major 5，Minor 6，已修复 0。
