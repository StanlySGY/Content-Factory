# 项目宪法

本文档是 Content Factory 的最高项目约束。后续产品设计、架构设计、开发实现、Agent 编排、MCP 集成、Skill 扩展、测试、部署和文档交付必须遵守本文件。

## 项目使命

构建一个基于 Claude Code、Codex、Gemini、OpenCode 等 Agent 的 AI 内容工厂，将内容生产从一次性聊天交互升级为可追踪、可复用、可审查、可扩展的工程化系统。

## 产品定位

Content Factory 是面向内容生产、内容运营和 AI Agent 编排的工作台。

核心能力：

- 将内容需求转化为结构化生产任务。
- 通过多 Agent 分工完成研究、规划、生成、审查、发布准备。
- 沉淀可复用的工作流、插件、Skill、MCP 工具和文档资产。
- 保障内容生产过程具备可追溯性、可审计性和可迭代性。

## 设计原则

- **文档优先**：重要设计不得只存在于聊天上下文。
- **模块自治**：模块拥有清晰职责、输入、输出和依赖边界。
- **可组合**：Agent、插件、MCP、Skill、工作流必须可独立组合。
- **可替换**：外部模型、工具和服务不得与业务逻辑强绑定。
- **可观测**：关键任务、状态、错误、审查结果必须可追踪。
- **最小必要**：只实现当前明确需要的能力，避免过度抽象。

## 技术原则

- 使用清晰的分层架构隔离 UI、API、领域逻辑、工作流编排、基础设施集成。
- 业务核心不得直接依赖具体 Agent SDK、MCP Server 或第三方平台。
- 外部能力通过适配器、接口或插件注册机制接入。
- 配置、提示词、模型参数、工作流步骤必须外置化或数据化。
- 错误处理必须显式，禁止吞掉异常或返回含糊状态。
- 安全、权限、审计、限流、重试、超时必须作为基础能力设计。

## 代码原则

- 遵循 KISS、YAGNI、DRY、SOLID。
- 单一职责优先，禁止巨型函数、巨型组件、巨型服务。
- 命名必须表达业务含义，禁止使用模糊缩写。
- 公共逻辑必须抽象复用，禁止复制粘贴扩散。
- 业务规则必须集中管理，禁止散落在 UI、脚本或 Agent Prompt 中。
- 测试必须覆盖核心领域逻辑、工作流状态流转和外部适配器边界。

## 文档原则

- 所有阶段性设计成果必须写入 `docs` 目录。
- 开发前必须读取 `docs/README.md` 和相关专题文档。
- 修改架构、工作流、Agent、MCP、Skill、API、数据模型时，必须同步更新文档。
- 文档必须说明目标、边界、约束、决策和后续行动。
- 文档保持精简、可执行、可维护，禁止空泛描述。

## 插件原则

- 插件必须通过稳定接口接入，不得直接修改核心业务流程。
- 插件必须声明名称、版本、能力、输入、输出、权限和失败策略。
- 插件不得持有不可控全局状态。
- 插件之间不得互相隐式依赖。
- 插件失败不得导致核心系统不可恢复。

## Agent 原则

- Agent 是能力执行者，不是业务规则所有者。
- Agent 职责必须可描述、可限制、可替换。
- Agent 与业务逻辑之间必须通过任务协议、上下文包或适配器交互。
- 禁止 Agent 直接写死业务流程、产品规则或数据结构。
- 多 Agent 协作必须有明确的任务边界、输入输出、审查机制和冲突处理策略。
- Agent 输出必须经过校验、审查或人工确认后进入关键流程。

## MCP 原则

- MCP Server 是外部能力边界，不得承载核心业务规则。
- MCP 工具必须声明用途、输入、输出、权限、超时和错误语义。
- MCP 调用必须最小权限、最小上下文、最小数据暴露。
- 涉及外部网络、敏感数据、生产环境的 MCP 调用必须有明确授权。
- MCP 返回结果必须作为事实输入校验，不得无条件信任。

## 工作流原则

- 工作流必须显式定义阶段、状态、输入、输出、负责人和质量门禁。
- 工作流状态必须可持久化、可恢复、可审计。
- 每个阶段必须有明确完成标准。
- 自动化流程必须支持人工介入和回滚。
- 内容生成、审查、发布准备必须分阶段解耦。
- 禁止把一次性聊天记录当作工作流状态来源。

## 开发规范

- 新功能开发必须先确认相关文档是否存在；不存在时先补文档。
- 代码变更必须限制在需求范围内，禁止顺手重构无关模块。
- 新增模块必须定义职责、边界、依赖和测试策略。
- 新增公共接口必须说明契约、错误行为和兼容性要求。
- 修复缺陷必须优先定位根因，禁止只修表象。
- 合并前必须运行与变更范围匹配的测试、检查或人工验证。

## Git 规范

- Commit 使用 Conventional Commits。
- Commit 必须聚焦单一目的，禁止混合无关变更。
- 未经用户要求，不主动创建分支、提交或推送。
- 推送远端、重写历史、删除分支等操作必须遵守危险操作确认机制。
- 提交前必须检查工作区状态，确认只包含本次目标变更。

## 目录规范

- `docs/00-project`：项目使命、宪法、路线图、决策记录。
- `docs/01-product`：产品需求、用户场景、验收标准。
- `docs/02-architecture`：系统架构、模块边界、架构决策。
- `docs/03-database`：数据模型、存储、迁移、索引。
- `docs/04-agent`：Agent 角色、协议、协作与审查机制。
- `docs/05-mcp`：MCP Server、工具契约、权限边界。
- `docs/06-skill`：Skill 注册、路由、质量门禁与扩展规范。
- `docs/07-workflow`：内容生产、编排、状态流转。
- `docs/08-ui`：信息架构、页面、组件、交互和设计系统。
- `docs/09-api`：接口契约、鉴权、错误码、版本策略。
- `docs/10-development`：开发环境、编码规范、测试策略。
- `docs/11-deployment`：部署、环境配置、发布、回滚和运维。

## 禁止事项

- 禁止业务逻辑写死在 UI、脚本、Prompt、Agent 或配置文件中。
- 禁止 Agent 与业务逻辑强耦合。
- 禁止 MCP Server 承载核心业务规则。
- 禁止跨模块直接依赖内部实现。
- 禁止跳过文档直接开发关键功能。
- 禁止以聊天上下文替代项目文档、状态记录或决策记录。
- 禁止复制粘贴形成重复逻辑。
- 禁止无测试或无验证交付核心流程变更。
- 禁止无授权调用生产环境 API、发送敏感数据或执行破坏性操作。
- 禁止在代码中硬编码密钥、模型配置、外部端点、业务规则和权限策略。
- 禁止为了短期实现破坏模块边界、接口契约和可替换性。

## 生效规则

- 本文件自提交后生效。
- 后续所有开发、设计、审查和重构必须以本文件为约束。
- 如果实现方案与本文件冲突，必须先更新并说明宪法变更原因，再继续实施。
