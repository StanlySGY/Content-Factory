## 2. 产品目标

Content Factory 的目标是构建一个工程化 AI 内容生产系统，将基于 Claude Code、Codex、Gemini、OpenCode 等 Agent 的内容生产过程，从一次性聊天协作升级为可配置、可追踪、可复用、可审查、可扩展的内容工厂。

### 2.1 核心目标

- 将内容需求转化为结构化任务、上下文包和可执行工作流。
- 支持多个 Agent 按角色分工完成研究、规划、撰写、审查、修订和发布准备。
- 沉淀可复用的内容模板、工作流、插件、Skill、MCP 工具和项目文档。
- 保证内容生产全过程具备状态追踪、版本记录、质量门禁和人工确认机制。
- 通过清晰模块边界，确保 Agent、MCP、Skill、插件与业务规则解耦。

### 2.2 成功标准

- 用户可以从一个内容需求创建完整生产任务。
- 系统可以按预设工作流调度不同 Agent 完成阶段性产出。
- 每个阶段的输入、输出、状态、责任方和审查结论可追踪。
- 内容资产、提示词、上下文、审查记录和最终稿可复用。
- 新 Agent、新 MCP 工具、新 Skill 可以通过插件化方式接入。

## 3. 产品定位

Content Factory 是面向内容生产、内容运营和 AI Agent 编排的工作台。

它不是单一聊天工具，也不是单一内容编辑器，而是将内容生产流程中的需求管理、资料研究、任务拆解、Agent 协作、内容生成、质量审查和发布准备组织为可持续运行的系统。

### 3.1 产品边界

包含：

- 内容生产任务管理。
- Agent 角色与任务编排。
- 工作流定义与执行追踪。
- 内容资产、上下文、模板和审查记录管理。
- MCP、Skill、插件的接入与治理。
- 内容生产过程中的质量门禁和人工确认。

不包含：

- 替代所有第三方发布平台。
- 将业务规则写死到某个 Agent 或 Prompt 中。
- 直接绑定单一模型供应商或单一 Agent 工具。
- 在无授权情况下自动调用生产环境 API 或外部发布渠道。

## 4. 目标用户

### 4.1 内容负责人

负责制定内容策略、确认内容方向、审批最终内容。

核心诉求：

- 快速创建内容需求。
- 查看内容生产进度。
- 审批关键产出。
- 复用高质量内容流程。

### 4.2 内容运营人员

负责日常选题、素材整理、内容生成、排期和发布准备。

核心诉求：

- 批量管理内容任务。
- 使用模板和工作流提升效率。
- 追踪每篇内容的来源、版本和审查记录。
- 减少重复沟通和手工整理。

### 4.3 AI 工作流设计者

负责设计 Agent 分工、工作流阶段、质量门禁和工具接入方式。

核心诉求：

- 配置可复用工作流。
- 定义 Agent 职责、输入输出和审查规则。
- 接入 MCP 工具、Skill 和插件。
- 调试工作流执行过程。

### 4.4 开发者与系统维护者

负责系统实现、扩展、部署和运维。

核心诉求：

- 清晰理解模块边界和接口契约。
- 安全接入外部 Agent、MCP 和插件。
- 维护可测试、可扩展、可观测的系统。
- 避免业务逻辑和外部工具强耦合。

## 5. 用户场景

### 5.1 从选题创建内容生产任务

用户输入内容主题、目标受众、发布渠道、语气风格和截止时间。系统生成结构化任务，并选择合适工作流。

期望结果：

- 创建内容任务。
- 自动生成需求摘要。
- 绑定默认工作流。
- 明确下一步执行阶段。

### 5.2 多 Agent 协作生产内容

系统将内容任务拆分为研究、提纲、初稿、审查、修订、发布准备等阶段，并分配给不同 Agent。

期望结果：

- 每个 Agent 只处理明确职责范围内的任务。
- 每个阶段生成可审查产出。
- 阶段之间通过结构化上下文传递信息。

### 5.3 人工审查与质量门禁

内容负责人在关键节点查看 Agent 输出，确认是否通过、退回修改或人工编辑。

期望结果：

- 记录审查意见。
- 保留版本差异。
- 阻止未经确认的内容进入发布准备阶段。

### 5.4 复用成熟内容工作流

用户将一条验证有效的内容流程保存为模板，用于后续同类内容生产。

期望结果：

- 保存工作流模板。
- 复用 Agent 分工和阶段定义。
- 支持按不同内容类型调整参数。

### 5.5 接入新 Agent 或工具

系统维护者将新的 Agent、MCP 工具或 Skill 接入平台，并声明能力、权限和失败策略。

期望结果：

- 新能力通过注册机制接入。
- 不修改核心业务流程。
- 可在工作流中按能力选择使用。

### 5.6 追溯内容生成过程

当内容出现质量问题或需要复盘时，用户查看需求、上下文、Agent 输出、审查意见和最终版本。

期望结果：

- 明确内容来源。
- 明确每次修改原因。
- 明确哪个阶段产生问题。
- 支持后续优化工作流。

## 6. 核心功能

### 6.1 内容任务管理

用于创建、编辑、查看和追踪内容生产任务。

功能要求：

- 创建内容任务，记录主题、目标、受众、渠道、风格、截止时间。
- 管理任务状态：草稿、待执行、执行中、待审查、需修订、已完成、已归档。
- 支持任务优先级、负责人、标签和内容类型。
- 支持任务历史记录和关键事件追踪。

### 6.2 需求结构化

将用户输入的自然语言需求转化为可执行的结构化任务定义。

功能要求：

- 提取内容目标、受众、渠道、格式、约束和验收标准。
- 生成任务摘要和缺失信息提示。
- 支持用户确认或补充需求。
- 禁止将一次性聊天上下文作为唯一需求来源。

### 6.3 工作流管理

定义和执行内容生产流程。

功能要求：

- 支持工作流模板。
- 工作流包含阶段、输入、输出、执行者、质量门禁和完成标准。
- 支持阶段状态持久化。
- 支持人工暂停、继续、退回和终止。
- 支持工作流执行日志。

### 6.4 Agent 编排

管理 Agent 角色、能力和任务分配。

功能要求：

- 定义 Agent 名称、角色、能力、输入输出和限制。
- 支持按阶段选择 Agent。
- 支持多 Agent 并行或串行协作。
- 支持 Agent 输出审查。
- Agent 不得直接拥有业务规则。

### 6.5 上下文包管理

为 Agent 提供最小必要、可审查的上下文输入。

功能要求：

- 生成阶段上下文包。
- 记录上下文来源和版本。
- 控制敏感信息暴露。
- 支持上下文复用和归档。

### 6.6 内容资产管理

管理内容生产过程中的中间产物和最终产物。

功能要求：

- 保存研究材料、提纲、初稿、修订稿、最终稿。
- 支持版本记录。
- 支持资产标签和检索。
- 支持关联任务、工作流阶段和 Agent 输出。

### 6.7 质量审查

确保内容在进入下一阶段前满足质量要求。

功能要求：

- 支持自动检查和人工审查。
- 审查项包括事实准确性、结构完整性、风格一致性、渠道适配性、风险内容。
- 支持通过、退回、人工修改、终止。
- 审查结果必须记录。

### 6.8 MCP 工具接入

接入检索、浏览器、文件、外部系统等 MCP 工具能力。

功能要求：

- 管理 MCP Server 与工具清单。
- 声明工具权限、输入、输出、超时和错误语义。
- 在工作流阶段中按能力调用工具。
- 对外部网络、敏感数据和生产环境调用执行授权控制。

### 6.9 Skill 管理

管理可复用命令、质量门禁和领域能力。

功能要求：

- 注册 Skill 名称、用途、触发条件和输出格式。
- 支持工作流阶段引用 Skill。
- 支持质量门禁 Skill。
- Skill 不得绕过项目宪法和文档原则。

### 6.10 插件系统

通过插件扩展内容类型、渠道、工具和工作流能力。

功能要求：

- 插件声明名称、版本、能力、权限、输入输出和失败策略。
- 插件通过稳定接口接入。
- 插件不得直接修改核心业务流程。
- 插件失败必须可隔离、可记录、可恢复。

### 6.11 仪表盘与可观测性

提供任务、工作流、Agent、工具调用和质量状态视图。

功能要求：

- 展示任务列表和状态。
- 展示工作流执行进度。
- 展示阶段输出和审查结果。
- 展示 Agent 执行记录和失败原因。
- 展示 MCP 调用记录和风险提示。

## 7. 功能优先级

### 7.1 P0：MVP 必须实现

- 项目文档体系。
- 项目宪法。
- 产品需求文档。
- 内容任务基础模型。
- 工作流阶段模型。
- Agent 角色模型。
- 人工可执行的内容生产流程定义。
- 阶段产出和审查记录。
- 最小可用的文档驱动开发流程。

验收标准：

- 用户可以根据文档创建内容任务并按阶段推进。
- 每个阶段的产出可以被记录和审查。
- Agent 角色与业务规则边界清晰。
- 文档覆盖产品、架构、Agent、工作流的基础定义。

### 7.2 P1：核心自动化

- 需求结构化。
- 工作流模板管理。
- Agent 分配规则。
- 上下文包生成。
- 内容资产版本管理。
- 基础质量门禁。
- MCP 工具清单与调用记录。

验收标准：

- 用户可以从自然语言需求生成结构化任务。
- 系统可以根据模板推进内容生产阶段。
- 每次 Agent 和 MCP 调用都可追踪。

### 7.3 P2：协作与扩展

- 多 Agent 并行协作。
- 插件注册与隔离。
- Skill 路由与质量门禁自动化。
- 内容模板库。
- 审查规则配置。
- 多渠道发布准备。

验收标准：

- 新 Agent、Skill、MCP 或插件可以在不修改核心业务逻辑的情况下接入。
- 内容生产流程可以按内容类型复用和调整。

### 7.4 P3：规模化与智能优化

- 工作流执行分析。
- 内容质量趋势分析。
- Agent 效果评估。
- 成本、耗时和失败率分析。
- 自动推荐工作流优化方案。
- 团队权限和多租户能力。

验收标准：

- 用户可以基于数据优化内容生产流程。
- 系统可以支持团队级内容工厂运行。

## 8. 未来规划

### 8.1 MVP 阶段

目标：建立文档、模型和人工可执行工作流基础。

重点：

- 完成核心文档体系。
- 明确产品、架构、Agent、MCP、Skill、工作流边界。
- 设计任务、阶段、产出、审查记录等基础模型。

### 8.2 Alpha 阶段

目标：实现基础系统闭环。

重点：

- 提供任务创建与阶段追踪界面。
- 支持工作流模板。
- 支持基础 Agent 执行记录。
- 支持内容资产版本管理。

### 8.3 Beta 阶段

目标：实现多 Agent 自动化协作。

重点：

- 接入 Claude Code、Codex、Gemini、OpenCode 等执行能力。
- 引入 MCP 工具治理。
- 引入自动质量门禁。
- 建立插件注册机制。

### 8.4 生产化阶段

目标：支持团队稳定运行。

重点：

- 完善权限、审计、安全与运维能力。
- 支持多项目、多团队、多工作流。
- 支持指标分析与工作流优化。
- 支持发布准备与外部平台集成。

## 9. 非功能要求

### 9.1 可维护性

- 核心业务逻辑必须与 Agent、MCP、Skill、插件解耦。
- 模块必须有明确边界和接口契约。
- 文档必须随设计和实现同步更新。

### 9.2 可扩展性

- 新 Agent、新 MCP 工具、新 Skill、新插件必须可注册接入。
- 工作流必须可配置、可复制、可版本化。
- 内容类型和发布渠道必须可扩展。

### 9.3 安全性

- 敏感数据不得默认暴露给 Agent 或 MCP 工具。
- 外部网络、生产环境 API、发布动作必须显式授权。
- 插件和工具调用必须具备权限边界。

### 9.4 可观测性

- 任务状态、阶段状态、Agent 输出、MCP 调用、审查结果必须可追踪。
- 失败必须记录原因、上下文和恢复建议。
- 关键操作必须具备审计记录。

### 9.5 可恢复性

- 工作流必须支持暂停、恢复、退回和终止。
- Agent 或插件失败不得导致任务状态不可恢复。
- 中间产物必须持久化，避免依赖聊天上下文。

## 10. 关键约束

- 后续开发必须遵守 `docs/00-project/project-constitution.md`。
- 设计成果必须落地到 `docs` 目录。
- 不允许以聊天上下文替代需求、设计、状态或决策记录。
- 不允许业务逻辑写死在 Agent、Prompt、MCP、Skill、插件或 UI 中。
- 不允许跨模块直接依赖内部实现。
- 不允许跳过文档直接开发关键功能。
